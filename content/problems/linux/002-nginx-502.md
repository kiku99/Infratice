## 상황

현재 웹 서버에 접속하면 502 Bad Gateway 에러가 발생합니다. Nginx를 리버스 프록시로 사용하고 있으며, 백엔드 애플리케이션으로 요청을 전달하는 구조입니다. 제공된 설정 파일과 로그를 분석하여 원인을 찾고 해결하세요.

## 데이터

### nginx.conf

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

### error.log

```log
2024/01/15 10:23:45 [error] 1234#0: *1 connect() failed (111: Connection refused) while connecting to upstream, client: 192.168.1.100, server: example.com, request: "GET / HTTP/1.1", upstream: "http://127.0.0.1:3000/", host: "example.com"
2024/01/15 10:23:46 [error] 1234#0: *2 connect() failed (111: Connection refused) while connecting to upstream, client: 192.168.1.101, server: example.com, request: "GET /api/health HTTP/1.1", upstream: "http://127.0.0.1:3000/api/health", host: "example.com"
```

### systemctl status app 출력

```bash

     Active: failed (Result: exit-code) since Mon 2024-01-15 10:20:00 UTC

```

## 해설

### 원인 분석

Nginx의 `error.log`에서 `Connection refused` 에러가 발생하고 있습니다. 이는 upstream 서버(포트 3000)에 연결할 수 없다는 뜻입니다.

`systemctl status` 결과를 보면 백엔드 애플리케이션이 `EADDRINUSE` 에러로 시작에 실패했습니다. 포트 3000을 이미 다른 프로세스가 점유하고 있어 앱이 바인딩하지 못한 것입니다.

### 해결 방법

```bash
# 1. 포트 3000을 점유 중인 프로세스 확인
ss -tlnp | grep :3000
# 또는
lsof -i :3000

# 2. 해당 프로세스 종료 (PID를 확인 후)
kill -9 <PID>

# 3. 애플리케이션 재시작
sudo systemctl restart app

# 4. 정상 실행 확인
sudo systemctl status app
curl -I http://127.0.0.1:3000/api/health
```

### 실무 팁

`EADDRINUSE` 에러는 이전 프로세스가 비정상 종료되어 포트를 놓지 않았을 때 흔히 발생합니다. `systemd` 서비스 파일에 `Restart=on-failure`와 `RestartSec=5`를 추가하면 자동 복구가 가능합니다. 근본적으로는 graceful shutdown을 구현하여 SIGTERM 시그널을 올바르게 처리하는 것이 중요합니다.
